# PRD-000018-BUG

## Short Name

Fix Relative Image Rendering

## Goal

Make valid Markdown image references render correctly in Quill's application render view.

## Context

Relative image references currently do not render correctly in the rendered Markdown view. This breaks expected Markdown behavior for image content and creates an inconsistency between valid source Markdown and what the application shows to the user.

## Scope

In:

- make valid Markdown image references display in the application render view
- determine how relative image paths should resolve from the currently loaded Markdown document
- reuse the same rendering behavior in browser mode when practical
- provide a placeholder in browser mode when the same rendering logic cannot safely or cleanly apply there

Out:

- broad media-management features beyond Markdown image rendering
- remote image-policy redesign unless it is required to support valid existing behavior
- unrelated render-pane refactors not needed to restore correct image display

## Plan

- Reproduce the current failure with a concrete relative-path markdown sample, including `![Quill workflow lifecycle](./assets/workflow-lifecycle.png)`, while a markdown file is loaded from disk in the desktop app.
- Confirm how image URLs are currently emitted by the markdown renderer and where relative paths are being resolved today.
- Define desktop rendering so Markdown image sources are resolved relative to the currently loaded markdown document when Quill knows that document's filesystem path.
- Preserve existing support for already-valid absolute and data-image sources unless the image work requires a narrowly scoped sanitizer adjustment.
- Pass the current document location or a resolved base path into the render pipeline so image rendering can distinguish between desktop file-backed documents and browser-only sessions.
- Reuse the same resolution behavior in browser mode only when Quill has enough file context to produce a safe, stable image source.
- When browser mode does not have a trustworthy filesystem base path, render a visible placeholder treatment instead of a broken image so the markdown remains understandable and the limitation is explicit.
- Keep the implementation slice narrow to markdown image rendering in the preview/render pane; do not add image management, uploads, or unrelated preview refactors.
- Record any decision about placeholder copy, styling hook, or path-normalization edge cases in `Audit` if implementation reveals additional within-scope clarification.

## Acceptance Criteria

- In the desktop app, a markdown document loaded from disk that contains `![Quill workflow lifecycle](./assets/workflow-lifecycle.png)` renders an actual image in the preview pane when that file exists relative to the markdown document.
- Relative image paths are resolved from the directory of the currently loaded markdown document, not from Quill's app HTML location or an unrelated browser page URL.
- Existing valid absolute image sources that Quill already supports, including `file:` URLs and current data-image snippets, continue to render after the change.
- The rendered output keeps the markdown-provided alt text on the image element.
- Browser mode reuses the same rendering behavior only when Quill can determine a safe equivalent base path from the active document context.
- When browser mode cannot safely resolve the relative path, Quill renders a clear placeholder in the preview pane instead of a silently broken image element.
- The browser fallback does not claim the image rendered successfully when Quill lacks the path context needed to resolve it.
- The change does not break unrelated markdown rendering such as ordinary links, inline code, tables, or existing preview editing behavior.

## Verification

- Verify the renderer output with targeted code-level checks around markdown image HTML generation for:
  - a relative image path such as `./assets/workflow-lifecycle.png`
  - an absolute `file:` image path
  - an existing data-image snippet
- Verify a desktop file-backed markdown sample end to end in Quill using a real markdown file plus sibling `assets/` image so the preview pane proves the relative-path case visually.
- Verify browser-mode behavior separately for the same markdown sample and confirm either:
  - the image renders when Quill has equivalent safe path context, or
  - the placeholder appears when that context is unavailable
- Confirm the resolved image source is based on the active markdown document location rather than `window.location.href` alone.
- Run regression checks that the preview pane still renders existing non-image markdown content correctly after the image-path change.
- Capture verification evidence as a combination of command output or harness assertions for renderer logic plus preview-visible confirmation for the desktop relative-path sample and the browser fallback behavior.

## Next Step

Move this item to `01 - Plan` and define the relative-path resolution rules, application rendering behavior, browser-mode fallback behavior, and regression-check approach.
