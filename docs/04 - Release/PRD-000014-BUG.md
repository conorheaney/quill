# PRD-000014-BUG

## Short Name

Fix Table Cell Link Rendering

## Goal

Make the render pane display markdown links correctly inside table cells, including same-document anchor links that should navigate to the matching heading within the current rendered markdown.

## Context

The original report was handled incorrectly. A prior change focused on making Windows-style file-path links render as anchors, but that did not address the real user-facing bug. The actual issue is with internal same-document links inside table cells, such as `[Make Source-Search Tooling Explicit](#6-make-source-search-tooling-explicit)`. In the render pane, clicking that link should jump to the matching heading inside the current rendered markdown. Instead, the app appends the fragment to the current URL and loads the default document, which is incorrect preview behavior.

The table-linking issue is now resolved, but a new regression has been observed in general markdown rendering. When href-like markup appears inside backticks, everything from the start of the href onward is rendered as a link. In the user's supplied verification bullets, the trailing `...` inside backticks became part of the rendered link instead of remaining literal inline-code text. That regression must be treated as part of this same PRD before the item can stay in `Test`.

## Scope

In:

- reproduce the incorrect same-document anchor behavior for markdown links inside render-pane table cells
- identify whether the failure happens in anchor generation, click handling, slug matching, or render-pane navigation
- fix the render path so same-document anchor links inside table cells navigate within the current rendered markdown pane
- verify that link text, target headings, and surrounding table structure all behave correctly after the fix
- reproduce and fix the general-markdown regression where backticked href-like content causes the rest of the line to render as a link

Out:

- unrelated external-link behavior unless it is directly required for the fix
- broader table-authoring improvements
- unrelated markdown parser refactors

## Example Reproduction Markdown

```md
| 6 | [Make Source-Search Tooling Explicit](#6-make-source-search-tooling-explicit) | pending | high | Name the expected search workflow and tooling so source discovery is consistent across runs. |
```

When the user clicks that link in the render pane, Quill should scroll or jump to the `# 6 Make Source-Search Tooling Explicit` heading in the current rendered markdown view. The current behavior instead mutates the browser URL and loads the default document, which breaks in-document navigation.

## Proposed Behavior

- Same-document markdown anchor links inside table cells should render as normal anchor elements in the render pane.
- The visible link text should remain intact inside the affected cell.
- Rendering the link should not break the surrounding row or column structure.
- Clicking a `#...` anchor link in a rendered table cell should jump to the matching heading in the current rendered markdown pane instead of navigating away or reloading the default document.
- Inline-code spans containing href-like text should stay literal code text and must not cause surrounding prose on the same line to become part of a rendered link.

## Plan

- Reproduce the failure against the live render-pane navigation path using a reduced markdown sample with a table-cell `#...` anchor and a matching heading.
- Confirm whether the failure comes from missing heading IDs, incorrect anchor href generation, or missing same-pane click interception.
- Apply the smallest fix that restores in-document navigation for rendered same-document anchors while preserving existing external-link behavior.
- Re-verify both table and non-table same-document anchors before promoting this item again.
- Reproduce the new backticked-href regression in general markdown and isolate whether the link parser is running before inline-code protection in a way that captures too much trailing text.

## Acceptance Criteria

- A markdown table containing at least one `[label](#matching-heading)` cell renders that cell with clickable linked text in the render pane.
- The rendered row keeps the expected number of cells after link rendering is applied.
- Clicking the rendered table-cell anchor jumps to the matching heading inside the current rendered markdown pane.
- The click does not reload the default document or navigate away from the current rendered markdown content.
- Links in table cells behave consistently with same-document anchor links rendered in non-table markdown content.
- In general markdown, href-like text surrounded by backticks remains inline code only and does not turn the rest of the line into a link.

## Implementation Notes

- The prior `file:///...` sanitizer change was a partial and mis-scoped response to the original bug report, so this item must stay in `Implement` until the live same-document navigation behavior is correct.
- Investigate both heading slug generation and render-pane anchor click handling, because correct HTML alone is not sufficient if clicks still escape the current preview context.
- The likely root cause is now narrowed to two cooperating issues: the markdown renderer was converting `#...` links into normal browser anchors, and the preview pane was not intercepting those clicks to route them through Quill's existing `scrollToHeading()` behavior.
- The item must be demoted from `Test` because the newly reported backticked-href regression means the current implementation still has unresolved markdown-link parsing behavior outside the table case.
- The new general-markdown regression was caused by parser order: markdown links were tokenized before inline-code spans were protected, so backticked link-like content could still be interpreted as a real link.
- The prior parser-order fix was incomplete because it still emitted raw HTML inside inline-code spans. In rendered HTML, browser parsing could then treat text like `<a href="#x">...` inside `<code>` as a live anchor, which matches the user-provided screenshot evidence.
