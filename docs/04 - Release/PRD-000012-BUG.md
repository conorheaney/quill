# PRD-000012-BUG

## Short Name

Fix Escaped Pipe Table Rendering

## Goal

Make the Markdown render pane display table rows correctly when a cell contains literal pipe characters that are part of the cell content rather than real column separators.

## Context

The render pane currently shows extra table cells when a table cell includes the standard Markdown `|` character as part of the text content. This makes valid table content render incorrectly and can visually corrupt the table structure even when the source Markdown is intended to keep that pipe inside a single cell.

## Scope

In:

- reproduce the render-pane table formatting issue with Markdown tables that include literal pipe characters inside cell content
- update the relevant Markdown table parsing or rendering logic so literal in-cell pipes do not create unintended extra columns
- verify that affected tables render with the expected number of cells and preserved content

Out:

- general table feature expansion beyond this bug
- non-table Markdown rendering changes unless they are directly required for the fix
- editor-side table authoring improvements

## Example Reproduction Markdown

```md
| 6 | Inconsistency | done | Align segmentation table columns between skill and template appendix â€” standardized to `Segment \ Market Share \ [Revenue/Value] \  [CAGR] \ [Notes]` with note that columns adapt per subsection | SKILL L329, TEMPLATE L317 |
| 7 | Inconsistency | done | Align companies table columns â€” template appendix now matches skill and template Section 6.2: `Company \ | Headquarters \ | Key Products/Platforms \ | Market Position \ | Recent Activity` | SKILL L331, TEMPLATE L318, L164 |
```

The second row is a concrete example of the current failure mode because the literal pipe characters inside the backticked table-column text are content, not real column separators, but the render pane currently treats them as extra cells.

## Proposed Behavior

- Render-pane tables should keep the original column count when a cell contains literal pipe characters as part of its content.
- Backticked inline code inside table cells should preserve literal pipe characters as text content rather than splitting the row into extra cells.
- The supplied reproduction rows should render as five-column rows, with the full description text kept inside the fourth cell.

## Plan

- Reproduce the issue against the supplied rows, harden the shared table parsing path, and verify the corrected cell counts in both parser output and rendered HTML.

## Acceptance Criteria

- The reproduction example in this PRD renders without creating extra cells in the second row.
- Table parsing or rendering correctly distinguishes real column separators from literal pipes that belong to cell content.
- The render pane shows the expected five cells for both reproduction rows while preserving the visible inline-code text.

## Implementation Notes

- Start by tracing the shared Markdown table parsing path used by the render pane and identify where raw pipe splitting happens.
- Treat this as a parser-hardening bug fix, not a table feature redesign.
- Verify the behavior against both plain escaped-pipe content and inline-code content that includes pipe characters, because the current reproduction includes inline code.
